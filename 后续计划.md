分阶段计划
Phase 1：IRTDKT 模块移植到后端
目标： 将 IRTDKT 的 Python 代码作为服务嵌入 FastAPI 后端

工作内容：

复制核心模块 — 将以下文件复制到 backend/app/services/adaptive/：

irt_rasch.py → 核心 IRT 模型
controller.py → 自适应控制器
dkt_light.py → 轻量知识追踪（优先使用，无需 PyTorch）
eval.py → 评估指标
创建题库映射 — 为每个游戏定义 item_id（难度锚点），因为 BrainTraining 目前用 difficulty_level（浮点数）而 IRT 需要离散的 item_id：

数箱子：按箱子数量范围分段，如 box_easy/box_mid/box_hard
算式回溯：按 n_back_level 映射，如 nback_1/nback_2/nback_3/nback_4
石头剪刀布：按指令组合难度分段
数据适配层 — 写一个 irt_data_adapter.py，从 game_interactions 表中读取数据并转换为 IRT 所需的 DataFrame 格式（补充 past_wrong_ratio、sum_right 等派生字段）

依赖确认 — 确认 requirements.txt 中已有 numpy、pandas、scikit-learn，如没有则添加

Phase 2：后端新增 IRT 服务层
目标： 创建 irt_service.py 管理模型的训练、持久化和在线推理

工作内容：

模型管理：

启动时：如存在已训练模型（pickle 文件），加载；否则从数据库读取历史数据进行冷启动训练
冷启动兜底：新用户历史不足时，用全局平均 theta（ability=0）作为初始估计
在线推理：

每次用户提交游戏动作后，调用 IRT 在线更新用户 theta（Newton 单步更新，已在 irt_rasch.py 实现）
返回下一题推荐难度
模型定期重训练：

每积累 N 条新交互数据后，异步重训练全局 IRT 模型（更新所有用户 theta 和题目难度）
持久化：

用户 theta 值存入数据库（在 users 表新增 irt_theta 字段，按游戏分别存储，或新建 user_ability 表）
模型参数序列化为 pickle 文件存入 backend/data/
Phase 3：替换 game_engine.py 的难度调整逻辑
目标： 用 IRT 控制器替换现有的基于准确率阈值的规则调难

现有逻辑（规则）：


accuracy < 0.6  → 降难度
accuracy > 0.85 → 升难度
替换为 IRT 逻辑：


theta = irt_service.get_theta(user_id, game)
target_difficulty = theta - logit(0.70)  # 目标正确率 70%
next_item = 从题库选最近 target_difficulty 的题
difficulty_level = item_id_to_difficulty(next_item)
改动范围：

game_engine.py：create_session() 中读取 IRT 推荐难度替代固定规则
games.py API：POST /games/{game}/sessions 无需改动接口签名，难度由服务层自动决定

风险与注意事项
风险	说明	应对策略
冷启动	新用户无历史数据，IRT 无法估计 theta	用全局默认 theta=0（中等难度）兜底，3 次作答后即可估计
数据量不足	题库过小导致 IRT 难度参数不稳定	每款游戏至少定义 5 个难度锚点，数据积累后再精细化
性能	每次作答触发 IRT 更新可能增加延迟	theta 在线更新是单步 Newton 法，计算量极小（<1ms）
兼容性	替换调难逻辑可能影响现有游戏会话	新旧逻辑并存，用 feature flag 控制，逐步切换
完成后的效果
每个用户每款游戏有独立的 能力值 theta，系统自动将题目难度调整至让正确率维持在 65-75% 的最优学习区间（Flow 状态）
Reports 页展示 能力成长曲线 和 Flow 命中率，让用户看到自己的认知提升
整个系统形成 感知 → 作答 → IRT 评估 → 难度调整 → 再感知 的完整智能体闭环
